# Development Standards

后续新增接口或核心方法时，统一遵守以下备注说明规范：

## 1. API 接口说明

- 每个新增或显著改造的接口函数，都需要补一段 docstring。
- docstring 至少说明：`用途`、`权限/调用边界`、`关键输入`、`关键输出`。
- 如果接口存在明显副作用，也要说明：例如会写库、触发重建索引、发送异步任务。

推荐格式：

```python
def create_xxx(...):
    """Create xxx for the current tenant and trigger downstream side effects."""
```

## 2. 核心方法说明

- 领域服务、基础设施适配器、编排入口中的核心方法必须补 docstring。
- docstring 至少说明：`职责`、`输入范围`、`返回结果`、`重要副作用`。
- 如果方法只是薄封装且语义非常直接，可以一行说明；否则要说明为什么存在这层抽象。

## 3. 注释使用原则

- 注释优先解释“为什么这样做”，不要重复代码字面含义。
- 只有在策略、权限、安全边界、兼容性、降级逻辑不够直观时再加行内注释。
- 避免把业务规则写死在零散注释里；规则本身应尽量落到命名、模型和方法边界上。

## 4. 检索与安全相关代码的额外要求

- 涉及检索融合、权限过滤、风险策略、脱敏、拒答的核心方法，必须说明它影响的是哪一层边界。
- 如果某个实现是开发态 fallback，而不是最终生产实现，要在 docstring 里明确写清楚。

## 5. 改造时的最低标准

- 新增接口：补接口 docstring。
- 新增核心服务方法：补方法 docstring。
- 改造后逻辑明显变化：同步更新原有 docstring，避免注释与代码不一致。
- 新增基础设施适配器：补类说明和关键方法说明。
